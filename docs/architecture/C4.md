# Architecture (C4)

Симулятор сценариев Warhammer 40k — веб-приложение с FastAPI-бекендом, HTMX/Alpine-фронтом и JWT-авторизацией.

**Обновлён:** 2026-05-16 — зафиксирована целевая архитектура canonical content pipeline: wiki `.md` остаётся authoring/source input, compiler нормализует данные в версионированные JSON artifacts с уникальными стабильными ids для каждого unit/weapon/detachment/stratagem/enhancement/rule, runtime и API читают только canonical JSON/registry, а display names остаются только UI/log labels. Добавлен data flow от wiki frontmatter до runtime-ready данных.

## Canonical Content Data Flow — wiki `.md` → runtime-ready JSON

Целевое правило: wiki больше не является runtime object model. Wiki — это authoring/source input. Runtime, validators, combat, roster builder, replay/result и AI должны получать данные через canonical content layer.

```mermaid

        RAW_ID["optional frontmatter id<br/>unit.orks.boyz<br/>weapon.orks.shoota"]

    subgraph Compile["2. Compile + validate"]
```

### Canonical identity contract

| Layer | ID type | Example | Mutability | Purpose |
|-------|---------|---------|------------|---------|
| Wiki/source object | `canonical_id` / `unit_id` / `weapon_id` | `unit.orks.boyz`, `weapon.orks.shoota` | Stable across renames/moves | Permanent content identity. |
| Runtime game object | `runtime_unit_id` | `game_abc:p1:unit.orks.boyz:0` | Created per battle | State maps, positions, wounds, engagement, kills. |
| Display label | `display_name` | `Boyz` | Can change/localize | UI/log text only; never a map key. |
| Persistence reference | canonical id + selected config | `unit.orks.boyz`, `squad_size=10` | Stable | Rosters/replays survive title changes. |

Non-negotiable boundaries:

- `display_name` is never used as the authoritative key for roster entries, unit state, replay attribution, damage, ownership, movement, or result summaries.
- `canonical_id` identifies the datasheet/object from content.
- `runtime_unit_id` identifies one concrete in-game instance.
- Replay/result payloads may include display labels as denormalized snapshots for readability, but must also persist canonical/runtime ids.

### Compiler responsibilities

1. Scan deterministic content roots: `wiki/units`, `wiki/factions`, `wiki/detachments`, `wiki/stratagems`, `wiki/enhancements`, `wiki/rules`.
2. Parse machine fields from YAML frontmatter only. Markdown body remains human documentation/lore/tactics.
3. Generate or verify stable ids:
   - Prefer explicit `id:` in frontmatter once migration starts.
   - Until every file has `id:`, derive ids deterministically from object kind + faction slug + file/title slug, then write migration reports for collisions.
4. Normalize to canonical JSON schemas: stats, points, squad metadata, weapons, keywords, tags, faction links, detachment links, rule references.
5. Validate required gameplay fields with no silent defaults for runtime-critical data.
6. Emit versioned JSON artifacts plus a manifest containing schema version, source paths, mtimes/content hashes, and collision/exception reports.
7. Runtime loaders read generated JSON, not raw wiki files. Tests may compile fixtures directly, but production runtime should not silently bypass the compiler.

### Architectural decisions for remediation

| Decision | Rule | Rationale |
|----------|------|-----------|
| Wiki remains source input, not runtime model | `.md` files compile to JSON before runtime consumption | Keeps Obsidian-friendly editing while making runtime deterministic/testable. |
| JSON artifacts are generated and versioned | Include `schema_version`, `content_hash`, `source_path` | Makes stale cache/content drift visible and reviewable. |
| Every content object has a stable id | Unit, weapon, detachment, enhancement, stratagem, faction, rule | Prevents rename/display collisions and supports durable rosters/replays. |
| Runtime ids are scoped per battle | `game_id:player:canonical_id:index` or equivalent | Same unit can appear multiple times and on both sides without collisions. |
| Display names are labels only | UI/log labels never key state maps | Fixes mirrored armies, localization, and result attribution bugs. |
| No unsafe pickle cache for content | JSON + manifest replaces pickle | Removes untrusted cache execution risk and stale object risk. |
