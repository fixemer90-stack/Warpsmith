# F4.12 — My Rosters: управление сохранёнными ростерами

## Цель (SMART)

**Страница `/my-rosters` где пользователь видит все свои сохранённые ростеры и может:**
- ✕ **Удалить** любой ростер (подтверждение → DELETE `/api/rosters/{id}`)
- ✏ **Редактировать** ростер — открывает Team Builder с предзагруженным ростером
- 👁 **Просмотреть** состав ростера (юниты, pts, detachment)
- 🎲 **Запустить симуляцию** — быстрый переход к Scenario Setup с выбранным ростером
- 📋 **Копировать** ростер (Duplicate — создаёт копию под новым именем)

**Критерий приёмки:**
1. Зайти на `/my-rosters`, увидеть таблицу всех своих ростеров
2. Нажать ✕ → подтверждение → ростер исчезает, счётчик обновляется
3. Нажать ✏ → переход на `/team-builder?edit={id}` с загруженными юнитами
4. Изменить состав → Save обновляет существующий ростер (PUT, не POST)
5. Free-юзер видит ограничение «1 из 1 ростеров» (премиум: «3 из ∞»)

### API — новые/модифицированные endpoints

| Method | Path | Auth | Описание |
|--------|------|------|----------|
| `GET` | `/api/rosters` | ✅ | Уже есть — список ростеров пользователя |
| `GET` | `/api/rosters/{id}` | ✅ | Уже есть — один ростер |
| `PUT` | `/api/rosters/{id}` | ✅ | **Новый** — обновить ростер (edit) |
| `DELETE` | `/api/rosters/{id}` | ✅ | Уже есть |
| `POST` | `/api/rosters/{id}/duplicate` | ✅ | **Новый** — копировать ростер |
| `POST` | `/api/rosters` | ✅ | Уже есть — создать новый |

### PUT /api/rosters/{id} — Request Body
```json
{
  "name": "Updated Army",
  "faction": "orks",
  "pts_limit": 2000,
  "detachment": "War Horde",
  "units": [
    {"unit_name": "Boyz", "squad_size": 20, "pts": 1600, "loadout": "", "weapons": []},
    {"unit_name": "Warboss", "squad_size": 1, "pts": 65}
  ]
}
```
Response: `{ "id": 118, "name": "Updated Army", ... }` (полный объект). `units` must be decoded to an array by `web/routes/api_rosters.py`; Team Builder edit mode calls `.map()` on this field.

### POST /api/rosters/{id}/duplicate
Создаёт копию ростера с суффиксом " (copy)".
Response: `{ "id": 119, "name": "My Army (copy)", ... }`

```python
@router.post("/rosters/{roster_id}/duplicate")
async def duplicate_roster(roster_id: int, user: User = Depends(get_current_user)):
    """Копировать ростер."""
    row = db.fetchone("SELECT * FROM rosters WHERE id = ?", (roster_id,))
    if not row or (row["user_id"] != user.id and not row["is_public"]):
        raise HTTPException(404, "Roster not found")

    new_name = row["name"] + " (copy)"
    new_id = db.insert("rosters", {
        "user_id": user.id,
        "name": new_name,
        "faction": row["faction"],
        "pts_limit": row["pts_limit"],
        "detachment": row["detachment"],
        "units": row["units"],
        "is_public": False,
    })
    return get_roster_by_id(new_id)
```

**`web/static/my_rosters.js`**:

```javascript
function myRosters() {
    return {
        rosters: [],
        tier: 'free',
        expanded: null,
        deleteTarget: null,

        async init() {
            await this.loadRosters();
            await this.loadTier();
        },

        async loadRosters() {
            try {
                const resp = await fetch('/api/rosters');
                if (resp.ok) {
                    const data = await resp.json();
                    this.rosters = data.rosters || [];
                    // Parse units JSON if string
                    this.rosters.forEach(r => {
                        if (typeof r.units === 'string') {
                            try { r.units = JSON.parse(r.units); } catch(e) { r.units = []; }
                        }
                    });
                }
            } catch (e) {
                console.error('Failed to load rosters:', e);
            }
        },

        async loadTier() {
            try {
                const resp = await fetch('/api/me');
                if (resp.ok) {
                    const user = await resp.json();
                    this.tier = user?.tier || 'free';
                }
            } catch (e) { /* ignore */ }
        },

        async duplicateRoster(id) {
            try {
                const resp = await fetch(`/api/rosters/${id}/duplicate`, { method: 'POST' });
                if (resp.ok) {
                    await this.loadRosters();
                } else {
                    alert('Failed to duplicate roster');
                }
            } catch (e) {
                alert('Network error');
            }
        },

        confirmDelete(roster) {
            this.deleteTarget = roster;
        },

        async deleteRoster() {
            if (!this.deleteTarget) return;
            try {
                const resp = await fetch(`/api/rosters/${this.deleteTarget.id}`, { method: 'DELETE' });
                if (resp.ok) {
                    this.deleteTarget = null;
                    await this.loadRosters();
                } else {
                    const err = await resp.json().catch(() => ({}));
                    alert(err.detail || 'Failed to delete roster');
                }
            } catch (e) {
                alert('Network error');
            }
        }
    };
}
```

## Edge Cases

| Ситуация | Поведение |
|----------|-----------|
| Free-юзер, уже есть 1 ростер → жмёт «+ New Roster» | Team Builder открывается, но Save показывает «Free tier limit: 1 roster» |
| Free-юзер удаляет единственный ростер | Счётчик: 0/1. Может создать новый |
| Редактирование чужого ростера через URL `?edit=999` | API возвращает 403 → показываем alert и редирект на /my-rosters |
| Удаление ростера, который используется в активной игре | Никаких проверок — удаляется. Реплей останется (game_state сохранён в таблице replays) |
| Дублирование публичного ростера | Создаётся приватная копия (`is_public: false`) |
| PUT с невалидными юнитами | Те же validation_errors что и при POST |
| Редактирование ростера без изменений → Save | PUT срабатывает, `updated_at` обновляется |

## Implementation Note — 2026-05-09

### Explicit Warlord selection in saved rosters

Team Builder and roster validation now preserve Warlord metadata in the roster unit JSON. This is required when a roster contains several Character/Warlord-capable units: the player must explicitly choose which model is the Warlord.

**Active files:**

- `web/static/team_builder.js` — Warlord UI state, save payload, edit-mode metadata hydration
- `web/routes/api.py` — unit list/detail metadata for `is_character` and `can_be_warlord`
- `web/routes/api_rosters.py` — roster validation/persistence and generated roster metadata
- `tests/test_rosters.py` — Warlord persistence and validation regressions

### Unit JSON contract

Roster units keep the gameplay metadata needed for validation and later simulation:

```json
{
  "unit_name": "Warboss",
  "squad_size": 1,
  "pts": 65,
  "is_warlord": true,
  "can_be_warlord": true,
  "is_character": true,
  "loadout": "",
  "nob_option": "",
  "weapons": []
}
```

Rule:

- zero or one Character/Warlord-capable unit: no ambiguity;
- multiple Character/Warlord-capable units: exactly one unit must have `is_warlord: true`;
- save and edit mode must not drop `is_warlord`, `loadout`, `nob_option`, `weapons`, or `pts`.

### UI behavior

Team Builder shows a crown (`👑`) control beside every eligible Character/Warlord-capable unit in the roster panel. When a second eligible Character is added, the previous implicit choice is cleared, an amber warning is shown, and Save stays disabled until the player clicks exactly one crown. The selected entry shows a yellow `Warlord` badge and the payload sends only that entry with `is_warlord: true`. In edit mode, roster entries are rehydrated from loaded unit metadata after `loadUnits()`, so old saved JSON plus current wiki/API metadata produce the same validation state as a newly built roster.
